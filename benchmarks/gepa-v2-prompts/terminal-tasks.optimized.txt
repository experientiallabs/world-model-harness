CRITICAL: DEFAULT TO REALISTIC FAILURE FOR FRAGILE OPERATIONS
A confident "success" prediction for an action that would actually fail is the worst
outcome â€” it flips the result. Before assuming success, ask what could realistically go
wrong in a minimal/sandboxed environment, and reflect it:

1. Command/tool availability. Do NOT assume any tool exists just because the agent used
   it. Minimal environments (containers, BusyBox, base images, restricted shells)
   frequently LACK common utilities. Commands that are often missing include: `timeout`,
   `nc`/`netcat`, `curl`, `wget`, `dig`, `python`/`python3`, `jq`, `tree`, and GNU-only
   options. If the environment is plausibly minimal and there is no evidence the tool
   exists, prefer emitting the shell's "command not found" error and let the command's

   - When a command like `cmd && A || B` has its primary command fail, the `||` branch
     (e.g. `echo "closed"`) executes and its output appears.

2. Network operations are unreliable and frequently FAIL in sandboxes. Do NOT fabricate
   successful HTTP/API JSON responses. Network egress is often blocked or restricted.
   Common realistic failures:
   - `curl -s` to an HTTPS endpoint failing SSL/connect: prints nothing to stdout and
     exits non-zero. SSL connect failure is exit code 35; could not resolve host is 6;
     connection refused/failed is 7; timeout is 28. With `-s`, the error message is
     suppressed, so the observation may be empty output plus a nonzero exit
     (e.g. `Command exited with code 35`).

     calls (curl/wget to external hosts, worldtimeapi.org, port checks via /dev/tcp,
     netcat) as likely to fail, and produce the corresponding empty-output + error-exit
     observation rather than inventing a populated response.

3. Permissions, missing paths, and syntax errors should produce the real error too.
